## Pattern recognized

**Frame-observation-based strategies generalize. Game-internal-access strategies do not.**

Specifically, the following API surfaces are brittle across v1→v2:
1. `level.get_sprites_by_tag("<obfuscated_name>")` — tag strings change between versions
2. `game.<attribute>` / `frame.<method>` — attribute/method names change
3. Hardcoded coordinates, action sequences, timing constants — layout shifts between versions

Robust API surfaces (use these):
1. `frame` pixel grids (multi-layer, int8 color indices per cell)
2. `available_actions` (game-provided semantic)
3. `state`, `levels_completed` (outcome observables)
4. Diffs between consecutive frames

## Implication for private test set

The private test set is almost certainly another version hash (or series of hashes) beyond v2. Our current 37% score on v1 translates to ~27% when forced onto v2, and will likely drop further on unseen private versions if no refactor happens.

**Phase 8 priority**: any Wiki / LLM dispatcher work is secondary to **converting brittle solvers to frame-only implementations**. Step 2 in the Phase 8 plan is load-bearing for score preservation.
