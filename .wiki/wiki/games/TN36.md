# TN36 — Programming Puzzle

> Bit-encoded action program: click bits to set a number, click play to execute, navigate player to goal; brittle solver passes v1 but collapses on v2.

## Observations

### Frame-level
- Single-layer 64×64 grid
- Discrete cells: "bit cells" arranged in a row that toggle on/off when clicked
- "Play" button region — pressing executes the encoded program as a series of movement steps
- Player sprite navigates on execution; kill zones end level
- Checkpoints exist (L6, L7) for multi-run programs

### Action effects
- `ACTION6(x, y)` on a bit cell toggles that bit
- `ACTION6(x, y)` on play button runs the current program
- Program = sum of enabled bits → maps to a movement step type

## Mechanics Hypothesis

- The game encodes a **tiny interpreter**: bits form an opcode; the player executes a short program from those opcodes
- Goal: reach target cell. Longer levels require multiple program runs with checkpoints caching state.

## Solution Pattern (current, brittle)

`strat_tn36_puzzle` in [`src/admorphiq/agent_ensemble.py`](../../../src/admorphiq/agent_ensemble.py):
1. Precompute target action code per level from game source
2. Call `frame.zpzcmabenn(val)` directly to set the program (bypasses bit-clicking entirely)
3. Click play
4. Repeat for multi-run levels with checkpoints

**Why it's brittle**: `frame.zpzcmabenn(val)` is a game-internal method. On v2 the method name is different (or removed) → AttributeError → 0/7.

## Proposed Frame-Only Refactor (Step 2a in Phase 8)

1. Identify bit cells: find a contiguous row of cells that change color on click (diff detection)
2. Identify play button: find the cell that triggers a player animation when clicked
3. Learn bit-to-action mapping: set single bits, run program, observe player motion vector
4. Plan: BFS over bit combinations and run counts to reach goal
5. No internal calls — only `ACTION6(x, y)` on detected cells

Expected performance post-refactor: v1 stays 7/7, v2 → ≥5/7.

## First Refactor Attempt — Findings (2026-04-21)

Added `strat_tn36_frame_only` in `src/admorphiq/agent_ensemble.py` as a Phase 8 Step 2a
fallback. It probes ACTION6 at rare-color cluster centroids and tries subsets + play
candidates. Test result:

- **v1**: still 7/7 via brittle `strat_tn36_puzzle` (runs first in dispatch).
- **v2** (`tn36-ef4dde99`): **0/7** — the fallback runs cleanly but fails to score.

Root cause identified by a 225-position probe sweep:

- **Every ACTION6 click produces diff = 1** (only a cursor-like pixel moves).
- **No single click causes level_up** — the mechanic genuinely requires a bit-sequence +
  play-button flow, not a single winning click.
- Simple subset enumeration (without knowing which cells are bits vs play vs learning
  the bit→direction map) is not sufficient.

### What is needed to actually clear v2 (new task)

1. **Bit-panel region detection**: find a row/column of equally-spaced small clusters
   whose click-diffs are spatially local.
2. **Play-button detection**: the single isolated cell whose click triggers a
   larger-scale animation (after bits are set).
3. **Bit semantics learning**: set single bits, press play, observe player vector; build
   bit→(dx, dy) map.
4. **BFS over bit patterns**: plan a sequence of `N_frames` bit values to reach goal.

See `[[../concepts/bit_encoding]]` for the algorithm and the new task filed in
`CLAUDE.md` for tracking.

## Second Refactor Attempt — Detection Works, Enumeration Still 0 (2026-04-21)

Expanded `strat_tn36_frame_only` with persistent-diff detection + bit-enumeration.
Diagnostic probes confirmed the layout on `tn36-ef4dde99`:

- **Bit row detected**: 6 cells at y=44, x ∈ {20, 24, 28, 32, 36, 40}, each click
  produces persist-diff ≈ 5 (stable toggle).
- **Play button detected**: cells at x ∈ {32..40}, y ∈ {52, 56}, click with bits set
  causes diff ≈ 70 (player animation).
- **Enumeration 0..63 failed to clear L1** (used 2000 actions).

Hypothesis: v2 L1 likely requires a **multi-frame program** like v1 (`[3, 3, 3, 3, 3]`
= 5 frames of value 3), not a single-value click. The single visible bit row may
represent one frame at a time, with implicit advancement between play clicks. Or the
game may cache multiple bit-row presses before the next play button click advances
the frame.

Further investigation needed:
- Does clicking play advance to a new frame (requiring multiple bit-row passes)?
- Is there a separate frame-selector UI element we haven't detected?
- Does the game show multiple bit rows at once in the v2 layout that we should detect?

This is deferred as Task #17-deep-dive. Current status: frame-only detection works,
scoring does not. v1 still 7/7 via brittle path.

### Lessons for Phase 8 refactor

- Detection is cheap; scoring requires understanding the full interaction model.
- Single-shot enumeration is insufficient when the game is a stateful interpreter.
- Future refactors should budget for a "game-model" phase between detection and
  exploitation.
