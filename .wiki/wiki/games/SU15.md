# SU15 — Vacuum Merge Puzzle

> 2048-style color merging via click-attract radius, with enemies as downgraders; solver reads internal fruit/enemy/goal lists, fails on v2.

## Observations

### Frame-level
- Multi-layer grid
- "Fruits" = colored sprites; same-color fruits merge on overlap to color+1 (like 2048)
- "Enemies" chase nearest fruit; contact downgrades the fruit by 1 (or removes if color 0)
- "Goal zones" = designated regions for delivering correct-color fruits
- Per-level goal spec: N fruits of color C plus optional enemy constraints

### Action effects
- `ACTION6(x, y)` creates a vacuum at (x, y) with radius ~8 px that pulls nearby fruits/enemies toward the click
- No simple actions matter here

## Mechanics Hypothesis

- Goal: by click-induced movement, arrange specific counts of specific colors in specific zones
- Two primitives available: **merge** (fruits) and **downgrade** (enemy hit)
- Level difficulty scales with target color (higher = more merges required)

## Solution Pattern (current, brittle)

`strat_su15_vacuum` in [`src/admorphiq/agent_ensemble.py`](../../../src/admorphiq/agent_ensemble.py):
1. Access `game.hmeulfxgy` → list of fruit sprites (positions, colors)
2. Access `game.peiiyyzum` → list of enemy sprites
3. Access `game.rqdsgrklq` → list of goal zones
4. Phase 0 — downgrade: suck over-colored fruits toward enemies
5. Phase 1 — merge: pair same-color fruits, suck them together
6. Phase 2 — deliver: suck target-color fruits to goal zones

**Why it's brittle**: `hmeulfxgy`, `peiiyyzum`, `rqdsgrklq` are obfuscated attribute names. On v2 they differ → AttributeError → 0/9.

## Proposed Frame-Only Refactor (Step 2b in Phase 8)

1. **Fruit detection**: cluster connected non-background pixels by color index; each cluster = one fruit candidate with (center_x, center_y, color, size)
2. **Enemy detection**: clusters whose color matches a reserved "enemy" palette (identified by negative behavior — they destroy, not merge). Learn this in first 5 clicks.
3. **Goal zone detection**: persistent background regions that never move; discovered by diffing multiple frames
4. Reimplement 3-phase plan using only the detected entities
5. Vacuum mechanic stays the same — only `ACTION6(x, y)`

Expected performance post-refactor: v1 stays 9/9, v2 → ≥6/9 (harder levels may need merge-order search).

### Frame-Only Scaffold (2026-04-21, WIP)

`strat_su15_frame_only` landed as an L1-L3 fallback that runs after `strat_su15_vacuum` returns 0 (typical v2 collapse path). Implemented:

- 4-connected flood-fill cluster detection over the 64×64 frame → `(cx, cy, color, size)` tuples (generalizable; will be lifted into `utils.py` once S5I5/RE86 reuse it).
- Probe click at `(2, 62)` to classify static vs dynamic colors by diffing cluster inventories.
- Greedy merge loop: for each same-color pair of dynamic clusters within ~2× vacuum radius, click the midpoint.

**Deliberately out of scope** (deferred to LLM hypothesis engine / BFS planner):
- Enemy avoidance and timing (L4+ require anticipating enemy movement).
- Multi-merge chaining to reach color `c3+` (needs lookahead).
- Hardcoded L7-L9 replacement (the existing brittle solver hardcodes these; a frame-only replacement is a Gate B target, not 2b).

Coverage target: v2 → 1-3 levels. The scaffold's main value is reuse, not raw score.
