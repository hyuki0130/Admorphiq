## 1. 개요: LLM Wiki란 무엇인가

LLM Wiki는 Andrej Karpathy가 제안한 **LLM 기반 개인 지식 베이스 구축 패턴**이다. 이 문서는 특정 구현체가 아닌 **아이디어 파일(idea file)**로서, 사용자가 자신의 LLM 에이전트(OpenAI Codex, Claude Code, OpenCode/Pi 등)에 복사하여 에이전트와 협업하며 구체적인 구현을 만들어 나가도록 설계되었다.

핵심 개념은 단순하다: **LLM이 점진적으로 구축하고 유지보수하는 영속적인 위키를 만들자.**

---

## 2. 기존 RAG 방식의 한계와 LLM Wiki의 차별점

### 2.1 기존 RAG의 문제점

대부분의 LLM-문서 연동 경험은 RAG(Retrieval-Augmented Generation) 방식을 따른다:

1. 파일 컬렉션을 업로드한다
2. 질의 시점에 관련 청크를 검색한다
3. 답변을 생성한다

NotebookLM, ChatGPT 파일 업로드, 대부분의 RAG 시스템이 이 방식이다. **이 방식은 작동하지만 근본적인 한계가 있다:**

- **매 질문마다 지식을 처음부터 재발견**해야 한다
- **축적(accumulation)이 없다** — 5개 문서를 종합해야 하는 미묘한 질문에도 매번 관련 조각들을 찾아 조합해야 한다
- **아무것도 구축되지(built up) 않는다**

### 2.2 LLM Wiki의 접근: 컴파일된 지식

LLM Wiki는 질의 시점에 원시 문서에서 검색하는 대신, **LLM이 점진적으로 영속적인 위키를 구축하고 유지보수**한다:

| 측면           | RAG                   | LLM Wiki               |
| -------------- | --------------------- | ---------------------- |
| 지식 처리 시점 | 질의 시점(query-time) | 수집 시점(ingest-time) |
| 교차 참조      | 매번 재발견           | 이미 존재              |
| 모순 탐지      | 없음                  | 이미 플래그됨          |
| 종합           | 매번 재수행           | 누적·갱신됨            |
| 복합적 질문    | 약함                  | 강함                   |

핵심 차이를 Karpathy는 이렇게 표현한다:

> **"위키는 영속적이고 복리적으로 성장하는 산출물(persistent, compounding artifact)이다."**

새로운 소스를 추가하면 LLM이 이를 단순히 인덱싱하는 것이 아니라, **읽고, 핵심 정보를 추출하고, 기존 위키에 통합**한다 — 엔티티 페이지를 업데이트하고, 주제 요약을 수정하고, 새로운 데이터가 기존 주장과 모순되는 곳을 기록하고, 진화하는 종합을 강화하거나 도전한다. 지식은 **한 번 컴파일되고 이후 최신 상태로 유지**된다.

## 3. 사용자와 LLM의 역할 분담

이 패턴에서 **사용자는 위키를 직접 작성하지 않거나 거의 작성하지 않는다.** 역할은 명확히 분리된다:

| 역할                                           | 담당 |
| ---------------------------------------------- | ---- |
| **소스 큐레이션**                              | 사람 |
| **탐색 및 질문**                               | 사람 |
| **요약, 교차 참조, 파일링, 부기(bookkeeping)** | LLM  |
| **분석 방향 설정**                             | 사람 |
| **일관성 유지보수**                            | LLM  |

Karpathy는 실제 작업 환경을 다음과 같이 묘사한다:

> "한쪽에 LLM 에이전트를 열고 다른 쪽에 Obsidian을 연다. LLM이 대화를 기반으로 편집하면, 나는 실시간으로 결과를 탐색한다 — 링크를 따라가고, 그래프 뷰를 확인하고, 업데이트된 페이지를 읽는다. **Obsidian은 IDE이고, LLM은 프로그래머이고, 위키는 코드베이스이다.**"

## 4. 적용 도메인

Karpathy는 이 패턴이 다양한 맥락에 적용 가능하다고 제시한다:

### 4.1 개인(Personal)

- 자신의 목표, 건강, 심리, 자기 개선 추적
- 일기 항목, 기사, 팟캐스트 노트를 파일링
- 시간에 걸쳐 자신에 대한 구조화된 그림을 구축

### 4.2 연구(Research)

- 수주~수개월에 걸쳐 주제를 깊이 파고듦
- 논문, 기사, 보고서를 읽고 점진적으로 종합 위키를 구축
- 진화하는 논제(thesis)를 유지

### 4.3 독서(Reading a book)

- 각 챕터를 순서대로 파일링
- 캐릭터, 테마, 플롯 스레드, 연결 관계를 페이지로 구축
- 책을 다 읽으면 풍부한 동반자 위키가 완성됨
- [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 같은 팬 위키를 개인적으로 구축하는 것과 유사 — 수천 개의 상호 연결된 페이지를 커뮤니티가 수년에 걸쳐 만드는 작업을, LLM이 교차 참조와 유지보수를 담당하며 개인이 수행 가능

### 4.4 비즈니스/팀(Business/team)

- LLM이 유지보수하는 내부 위키
- Slack 스레드, 회의록, 프로젝트 문서, 고객 통화를 소스로 공급
- 사람이 업데이트를 검토하는 루프 가능
- **팀에서 아무도 하고 싶지 않은 유지보수를 LLM이 수행**하므로 위키가 최신 상태 유지

### 4.5 기타

- 경쟁 분석, 실사(due diligence), 여행 계획, 수업 노트, 취미 심층 탐구
- **시간에 걸쳐 지식을 축적하고 흩어지지 않게 정리하고 싶은 모든 영역**

## 5. 아키텍처: 3계층 구조

```
┌─────────────────────────────────────────────┐
│              Schema (CLAUDE.md 등)           │  ← 구조·규칙·워크플로우 정의
├─────────────────────────────────────────────┤
│              Wiki (마크다운 파일들)            │  ← LLM이 소유·작성·유지보수
├─────────────────────────────────────────────┤
│              Raw Sources (원시 소스)          │  ← 불변(immutable), 진실의 원천
└─────────────────────────────────────────────┘
```

### 5.1 Raw Sources (원시 소스 계층)

- 사용자가 큐레이션한 소스 문서 컬렉션
- 기사, 논문, 이미지, 데이터 파일
- **불변(immutable)** — LLM은 읽기만 하고 절대 수정하지 않음
- **진실의 원천(source of truth)**

### 5.2 Wiki (위키 계층)

- LLM이 생성한 마크다운 파일들의 디렉토리
- 요약, 엔티티 페이지, 개념 페이지, 비교, 개요, 종합
- **LLM이 이 계층을 완전히 소유**
- LLM이 페이지를 생성하고, 새 소스 도착 시 업데이트하고, 교차 참조를 유지하고, 모든 것을 일관되게 유지
- 사용자는 읽기만; LLM이 쓰기를 담당

### 5.3 Schema (스키마 계층)

- LLM에게 위키의 구조, 규칙, 워크플로우를 지시하는 문서
- Claude Code의 `CLAUDE.md`, Codex의 `AGENTS.md` 등
- **핵심 설정 파일** — LLM을 범용 챗봇이 아닌 **규율 있는 위키 관리자**로 만드는 역할
- 사용자와 LLM이 도메인에 맞게 **함께 진화시킴**

## 6. 핵심 오퍼레이션

### 6.1 Ingest (수집)

새 소스를 원시 컬렉션에 넣고 LLM에게 처리를 지시한다. 예시 흐름:

1. LLM이 소스를 읽는다
2. 사용자와 핵심 시사점을 논의한다
3. 위키에 요약 페이지를 작성한다
4. 인덱스를 업데이트한다
5. 위키 전체의 관련 엔티티·개념 페이지를 업데이트한다
6. 로그에 항목을 추가한다

**단일 소스가 10~15개의 위키 페이지에 영향을 줄 수 있다.**

Karpathy의 선호 방식:

- 소스를 **한 번에 하나씩** 수집하고 관여를 유지
- 요약을 읽고, 업데이트를 확인하고, 무엇을 강조할지 LLM을 안내
- 그러나 **대량 일괄 수집**도 가능 (감독 수준은 사용자 재량)
- 자신의 워크플로우를 개발하고 스키마에 문서화하여 향후 세션에 활용

### 6.2 Query (질의)

위키에 대해 질문한다. LLM이 관련 페이지를 검색하고, 읽고, 인용과 함께 답변을 종합한다.

**답변의 형태는 질문에 따라 다양할 수 있다:**

- 마크다운 페이지
- 비교 테이블
- 슬라이드 덱 (Marp)
- 차트 (matplotlib)
- 캔버스

**핵심 인사이트: 좋은 답변은 위키에 새 페이지로 다시 파일링될 수 있다.**

요청한 비교, 분석, 발견한 연결 — 이것들은 가치 있으며 채팅 기록 속에 사라져서는 안 된다. 이 방식으로 **탐색 활동이 수집된 소스와 마찬가지로 지식 베이스에서 복리적으로 성장**한다.

### 6.3 Lint (건강 검사)

주기적으로 LLM에게 위키의 건강 상태를 점검하도록 요청한다. 점검 항목:

| 점검 항목                     | 설명                                    |
| ----------------------------- | --------------------------------------- |
| **모순(Contradictions)**      | 페이지 간 상충하는 주장                 |
| **낡은 주장(Stale claims)**   | 새로운 소스가 대체한 구 정보            |
| **고아 페이지(Orphan pages)** | 인바운드 링크가 없는 페이지             |
| **누락된 페이지**             | 언급되지만 자체 페이지가 없는 중요 개념 |
| **누락된 교차 참조**          | 연결되어야 하지만 연결되지 않은 관계    |
| **데이터 갭**                 | 웹 검색으로 채울 수 있는 정보 공백      |

LLM은 **조사할 새로운 질문과 찾아볼 새로운 소스를 제안**하는 데도 능하다. 이 과정이 위키가 성장하면서도 건강하게 유지되도록 한다.

## 7. 인덱싱과 로깅: 두 개의 특수 파일

### 7.1 index.md — 콘텐츠 지향적

- 위키 내 모든 페이지의 **카탈로그**
- 각 페이지는 링크, 한 줄 요약, 선택적 메타데이터(날짜, 소스 수) 포함
- 카테고리별 정리 (엔티티, 개념, 소스 등)
- **모든 수집(ingest) 시 LLM이 업데이트**
- 질의 응답 시 LLM이 **먼저 인덱스를 읽고** 관련 페이지를 찾은 뒤 상세 페이지로 진입
- **중간 규모(~100개 소스, ~수백 페이지)에서 놀랍도록 잘 작동**
- 임베딩 기반 RAG 인프라의 필요성을 회피

### 7.2 log.md — 시간순 기록

- **추가 전용(append-only)** 기록
- 무엇이 언제 발생했는지: 수집, 질의, 린트 패스
- 유용한 팁: 각 항목이 일관된 접두사로 시작하면 단순 유닉스 도구로 파싱 가능

```
## [2026-04-02] ingest | Article Title
```

```bash
grep "^## \[" log.md | tail -5   # 최근 5개 항목 조회
```

- 위키 진화의 타임라인 제공
- LLM이 최근 수행 작업을 이해하는 데 도움

## 8. 선택적 CLI 도구

위키가 성장하면 LLM이 위키를 더 효율적으로 운영할 수 있도록 **소형 도구 구축**을 고려할 수 있다.

### 검색 엔진

- **소규모**: index.md만으로 충분
- **대규모**: 적절한 검색 엔진 필요
- [**qmd**](https://github.com/tobi/qmd) 추천:
  - 마크다운 파일용 로컬 검색 엔진
  - 하이브리드 BM25/벡터 검색 + LLM 리랭킹
  - 모든 것이 온디바이스
  - CLI (LLM이 셸 아웃 가능) + MCP 서버 (LLM이 네이티브 도구로 사용 가능)

- 또는 LLM의 도움을 받아 **나이브 검색 스크립트를 직접 바이브코딩** 가능

## 9. 실용 팁과 도구 체인

### 9.1 Obsidian Web Clipper

- 웹 기사를 마크다운으로 변환하는 브라우저 확장
- 소스를 원시 컬렉션에 빠르게 추가하는 데 유용

### 9.2 이미지 로컬 다운로드

- Obsidian 설정 → Files and links → "Attachment folder path"를 고정 디렉토리로 설정 (예: `raw/assets/`)
- 설정 → Hotkeys → "Download attachments for current file" 검색 → 단축키 바인딩 (예: Ctrl+Shift+D)
- 기사 클립 후 단축키를 누르면 모든 이미지가 로컬 디스크에 다운로드
- LLM이 깨질 수 있는 URL 대신 이미지를 직접 조회·참조 가능
- **주의**: LLM은 인라인 이미지가 포함된 마크다운을 한 번에 읽을 수 없음 → 텍스트 먼저 읽고, 참조된 이미지를 별도로 조회하는 우회 방식 사용

### 9.3 Obsidian 그래프 뷰

- 위키의 형태를 보는 최적의 방법
- 무엇이 무엇과 연결되었는지, 어떤 페이지가 허브인지, 어떤 페이지가 고아인지 시각화

### 9.4 Marp

- 마크다운 기반 슬라이드 덱 형식
- Obsidian 플러그인 존재
- 위키 콘텐츠에서 직접 프레젠테이션 생성에 유용

### 9.5 Dataview

- 페이지 프론트매터에 대해 쿼리를 실행하는 Obsidian 플러그인
- LLM이 위키 페이지에 YAML 프론트매터(태그, 날짜, 소스 수)를 추가하면 Dataview가 동적 테이블과 목록 생성 가능

### 9.6 Git 버전 관리

- 위키는 **마크다운 파일의 git 저장소**에 불과
- 버전 기록, 브랜칭, 협업을 무료로 획득

## 10. 이 패턴이 작동하는 이유: 유지보수 비용의 혁신

### 10.1 인간 위키의 실패 원인

지식 베이스 유지보수에서 힘든 부분은 읽기나 사고가 아니다. **부기(bookkeeping)**이다:

- 교차 참조 업데이트
- 요약을 최신 상태로 유지
- 새로운 데이터가 기존 주장과 모순될 때 기록
- 수십 페이지에 걸친 일관성 유지

**인간은 유지보수 부담이 인지된 가치보다 빠르게 증가하기 때문에 위키를 포기한다.**

### 10.2 LLM의 구조적 우위

| 인간                        | LLM                         |
| --------------------------- | --------------------------- |
| 지루해진다                  | 지루해지지 않는다           |
| 교차 참조 업데이트를 잊는다 | 잊지 않는다                 |
| 한 번에 2-3개 파일 수정     | 한 번에 15개 파일 수정 가능 |
| 유지보수 비용 증가          | **유지보수 비용 ≈ 0**       |

위키가 유지되는 이유는 유지보수 비용이 거의 영(zero)에 가깝기 때문이다.

### 10.3 역사적 맥락: Vannevar Bush의 Memex

이 아이디어는 **Vannevar Bush의 Memex(1945)**와 정신적으로 관련된다:

- 문서 간 연상적 경로(associative trails)를 가진 개인적이고 큐레이션된 지식 저장소
- Bush의 비전은 웹이 실제로 된 것보다 LLM Wiki에 더 가까웠다: **사적이고, 능동적으로 큐레이션되며, 문서 간의 연결이 문서 자체만큼 가치 있는 것**
- Bush가 해결하지 못한 부분: **누가 유지보수를 하는가?**
- **LLM이 그것을 담당한다.**

## 11. 설계 철학: 의도적 추상성

이 문서는 **의도적으로 추상적**이다. 아이디어를 기술하지, 특정 구현을 기술하지 않는다.

**사용자가 결정할 사항들:**

- 정확한 디렉토리 구조
- 스키마 규칙
- 페이지 형식
- 도구 체인
- 소스 유형 (텍스트 전용이면 이미지 처리 불필요)
- 위키 규모 (작으면 검색 엔진 불필요)
- 출력 형식 (슬라이드 덱, 차트 등 필요 여부)

**올바른 사용 방법**: 이 문서를 LLM 에이전트와 공유하고, 함께 자신의 필요에 맞는 버전을 인스턴스화하는 것이다. 문서의 유일한 역할은 **패턴을 전달하는 것**이고, LLM이 나머지를 알아낼 수 있다.

## 12. 종합 평가

### 12.1 강점

- **복리적 지식 축적**: RAG의 근본 한계인 "매번 재발견" 문제를 구조적으로 해결
- **역할 분리의 명확성**: 인간 = 큐레이션/방향, LLM = 유지보수/부기
- **도구 불가지론(tool-agnostic)**: 특정 LLM이나 도구에 종속되지 않는 패턴
- **점진적 복잡성**: 최소한의 구성에서 시작해 필요에 따라 확장 가능
- **Git 기반 인프라**: 별도 인프라 없이 버전 관리, 협업, 롤백 가능

### 12.2 고려 사항

- **LLM의 정확성 의존**: 위키를 LLM이 전적으로 작성하므로, 환각(hallucination)이 위키에 고착될 위험 존재
- **스케일링 한계**: ~수백 페이지 이후 index.md 기반 탐색의 한계 → 검색 엔진 필요
- **비용**: 소스 하나당 10~15개 페이지를 업데이트하는 것은 상당한 토큰 소비를 의미
- **멀티유저 시나리오**: 팀 사용 시 동시 편집, 권한, 검토 프로세스에 대한 추가 설계 필요
- **LLM의 이미지 처리 제약**: 인라인 이미지를 한 번에 처리하지 못하는 현재 한계

### 12.3 핵심 통찰

이 패턴의 가장 강력한 통찰은 **"유지보수 비용의 비대칭성"**이다. 인간에게 유지보수는 가치보다 빠르게 증가하는 비용이지만, LLM에게 유지보수는 거의 무비용이다. 이 비대칭성 하나만으로 기존에 실패하던 개인 지식 관리 시스템이 지속 가능해진다. Karpathy는 이를 Vannevar Bush의 1945년 Memex 비전과 연결하며, 80년간 미해결이었던 "누가 유지보수하는가?"라는 질문에 대한 답이 드디어 도착했음을 시사한다.
